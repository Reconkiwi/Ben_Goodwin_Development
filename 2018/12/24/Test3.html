The LUFC Site - Y3S1

The LUFC News project was a rebrand of Alex Matthews’ fan made community/ social media group and website. I had a direct role in re-developing Alex’s website by creating a new Wordpress theme for him from scratch which included a login system and live chat room.

The Brief

During this semester we were required to organise or create our own briefs to complete over a 12 week period. As my development projects for my work placement had not begun yet, I decided to source an alternative project to work on for the rest of the semester rather than waiting for my work placement project to get up and running.

Alex Matthews, a client that I had worked with in my previous year of study (developing a prototype for his fan based Leeds Football news app), was looking to rebrand his community and identity starting with his social media platforms and eventually moving up to his website lufcnews.co.uk.

I was contacted after the rebrand of the company in the hope that I would build the site based on Alex’s LUFC News WordPress theme. Here is how the site looked before development started:

Alex had said to me that the website was no longer gaining consistent traffic and that the design had become outdated as he did not have the time or necessarily the skills to keep updating the templates.

It was clear that Alex’s website had become a little dated, so it was important for me to gain an understanding of precisely what he required from the sites redevelopment.

Initial Problems and Solutions

There were two main initial problems before development. Firstly, I had never developed using WordPress before, therefore, I had a fairly steep learning curve to take into account so that I could manage my time accordingly and I would need to read a few tutorials or speak to a WordPress specialist on my course as well to ensure I fully understood the WordPress theme development. Secondly, having looked through the code of Alex’s original theme I felt that it wouldn’t be possible to rebuild the site based off of his theme. This meant that I needed to discuss building a new theme from scratch with Alex as well as needing to find the quickest possible solution to building a WordPress theme from scratch. For this I needed to come up with a series of ideas:

Idea 1 - Building Pages and Directories from Scratch

Initially I had considered creating the pages, configurations and directories from scratch. This would have meant that I would have a better understanding of WordPress’ systems and setup. As well as this it would mean that all of the code on the theme would be mine which would help with development further down the line as everything would be setup in the same format.

Problems With Idea 1

The main issue that i faced with this idea was the time that it would take me to get a blank WordPress theme up and running. Alex was hoping to see progress within the first two weeks of development which meant that I didn’t have a large window to get some of the pages completed.

Idea 2 - A Course Mates Blank Template

The second idea was to build the site using a course mates blank WordPress theme. The benefits to this being that the blank template was being used commercially already, therefore, I was aware of the templates potential as I could see how it had been implemented into many sites.

Problems With Idea 2

The downside to this idea was that the blank template needed to be reformatted to ensure that it was not breaching any of the themes licensing agreements. This would have taken quite a lot of time and would have potentially caused more problems rather than solve any.

Idea 3 - Bones Theme

The third idea was to look into using the Bones WordPress theme. Bones is a HTML5, mobile first, blank theme used for rapid development. Bones also came with a free license meaning that I could use it for Alex’s site free of charge

Problems With Idea 3

My biggest problem with Bones was that I couldn’t get on with the documentation. This was a really large issue for me considering my lack of knowledge in Wordpress development.

Idea 4 - Underscores theme

Having looked into various solutions to my problem, and still being unsatisfied with my options, a masters student suggested Underscores to me. Underscores is a little like Bones, however, it aims to use minimal CSS which allows for developers to tear the base theme apart and completely rebuild it from the ground up. As well as this Underscores comes with a ready to go translation package and custom 404 pages to help get developers started.

The Final Solution

I ended up opting for Underscores rather than any of the other options, this was mainly due to the large amounts of documentation and tutorials based around the theme. I felt that this would be really beneficial to me as I could use these to increase my development efficiency.

Initial Development

Database

I began development by creating a database for our data to be collected in. During the first week, Jack worked alongside me to ensure that we both understood how the database would work and so that we both had a copy of the same table setup (Same names and types).

From our research we had found that we needed to set our ID row to ‘Primary’ as well as ‘AUTO_INCREMENT’, therefore, we implemented this into our database. ‘AUTO_INCREMENT’ is by default set to 1 and will increment by 1 for each new record. This is useful for the ID row as each visitor that is recorded in the database needs a unique ID. The Primary key is used to uniquely define a record. As there can only be one primary key in a table, it should be set to the most unique set of data within a record e.g. the ID so that each visitor can be defined.

We then created a TIMESTAMP for each record so that the silk mill can see the times in which different visitors attended the silk mill exhibitions.

Finally, we created the rest of the datasets that the silk mill needed to gather such as age, gender, country and exhibitions. For the prototype we limited the number of different exhibitions to 6 so that we could create a far fully fleshed out prototype.

Evaluation System - Version 1 Once Jack had mocked up a simple way of updating the database with information (through the virtual tour guide), it was my turn to begind developing a system that turned that data into graphs and charts.

To draw data from the database, we required a PHP document that connected our application to the MySQL database. This is then called at the start of each page of the application.

Having set up a connection to the database I then had to develop a way of pulling sets of data to implement into my Chart.JS charts. I began by creating a MySQL query that pulled all data from the ‘users’ database.

To set the data that I wanted to pull from the database I created an array of the rows that I needed. If the array is equal to the result, the query fetches the data from the database.

The result of this displays a string of data for each unique user. With the data pulled from the database, I was then able to call data.PHP in my Chart.JS page

I began my development by following a series of tutorials explaining how to import MySQL data into Chart.JS. Although, I quickly managed to create a very simple chart, I found it difficult to process and present the right data. Learning Chart,JS was not necessarily the problem, it was more that my MySQL queries were not sending the right information. I began troubleshooting my data.php file and messed around with my queries until I was able to represent the data that I wanted onto a graph.

I set the label dataset of the X-axis to take age from data.php. By doing this I was able to see each individual record across my X-axis with each exxhibit dataset along the Y-axis. However, the age dataset did not group people of similar ages. I experimented with my MySQL query and added ‘GROUP BY age’ to the end of this. This made visitors with the exact same age grouped onto my graph which was a slight improvement from what it previously was.

Unfortunately, I still felt as if the chart was not up to standard. I tried creating a MySQL query that set age ranges, however, I could not get the query to work. It wasn’t until the last version of the evaluation system that I was able to define age groups rather than individual ages.

After many failed attempts to improve the quality of my chart data, I decided to focus my work on developing another, more basic, chart ready for our first evaluation. I attempted to develop a doughnut chart that represented the total number of times each exhibit had been visited.

Safeguarding Information

The final piece of feedback I was given was to ensure that the evaluation software was protected so that only volunteers at the silk mill could access the data collected.

To do this, I created a basic password Javascript script that required a user to login to access the evaluation software. Although this system worked, I had noticed that anyone could find the password in the script through the inspect function on a webpage. This meant that the password was not secure at all. I decided to try and find a way of encrypting the password so that it would not be visible inside of the inspector. I began researching into htaccess and php login pages where passwords would be stored inside of databases.

The problem with this was that, I would have to backtrack my work and develop an entirely new login system which I felt would set my progress back. Eventually, I came across another way of hiding my password. The solution to my problem was to obfuscate my script. This method jumbled my JS code into a more obscure format. Although, this is not the most complex safeguarding system, I believed that this was complex enough for the prototype we were developing. I feel that this was an important addition to my project as data security is extremely important in today’s society.

Version 2

Having made these changes I ready for the crit review, the feedback was far more positive. However, it was noticed that I had not developed a redirectory system for my evaluation system. This meant that users could directly bypass the login page and have full access to the data collected through our application. I was extremely glad that this was pointed out as it was something that I had overlooked during my development.

Changes, Fixes and Development for Final Product Having considered the feedback during the week 10 crit revue. It was clear that I had to re-evaluate my safeguarding system. I began by looking at ways of redirecting my page unless a user had logged in.

I ended up developing a system using PHP where, through the use of sessions and defining ‘current’ and ‘last page’, I was able to check whether or not a use has accessed the page before. After this an ‘if’ statement is called that redirects the user from the evaluation page if they have not successfully logged in through the login page

I was really happy with how well this system worked. No matter how i tried to gain access to the page I would always be redirected back to the login page. I think that this really strengthened the safeguarding system I had in place. With this system now in place, and more time left to experiment with the project, I decided to have a look at other areas I could strengthen.

CSV Exporting

To strenthen my project even further and to improve the evaluation system for the volunteers at the silk mill, I decided to create a CSV exporting system, where volunteers could see all of the rows of data on the database and export them to an Excel document.

To create this, I developed a system that gathered the database data into an array and then echoed it into a table on the page. To export the data I created a form that posted the data to download.php. Download.php then downloads the data to a file called users.csv.

I think that this addition to the system will be really helpful for the volunteers at the silk mill as they will be able to have easy access to all of the data that our virtual guide will collect. As well as this I feel that it adds a very professional and complete feel to the project.
